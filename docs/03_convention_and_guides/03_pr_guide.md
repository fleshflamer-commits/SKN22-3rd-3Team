# 03. PR Guide

Pull Request(PR)는 코드 리뷰와 품질 관리를 위한 핵심 과정입니다.

## 1. PR 생성 규칙
- **작은 단위로 생성**: 하나의 PR은 하나의 논리적인 작업(이슈)만 포함해야 합니다. 너무 큰 PR은 리뷰하기 어렵습니다.
- **최신 상태 유지**: PR 생성 전 `main` 브랜치의 최신 내용을 병합(Merge 또는 Rebase)하여 충돌을 해결해야 합니다.

## 2. PR 제목 및 내용
PR 템플릿을 활용하여 내용을 충실히 작성합니다.

### 2.1 제목
커밋 메시지 컨벤션과 동일하게 작성합니다.
예: `feat: 메인 홈페이지 UI 개선`

### 2.2 내용 (Description)
- **작업 개요**: 무엇을 왜 변경했는지 설명합니다.
- **주요 변경 사항**: 핵심적인 구현 내용이나 디자인 변경점을 요약합니다.
- **관련 이슈**: 관련된 Jira 티켓이나 GitHub Issue 번호를 링크합니다 (e.g., `Closes #12`).
- **테스트 방법**: 리뷰어가 변경 사항을 어떻게 확인할 수 있는지 안내합니다.
- **스크린샷**: UI 변경이 있다면 전/후 스크린샷을 첨부합니다.

## 3. Code Review Process
- 리뷰어는 코드를 꼼꼼히 확인하고 건설적인 피드백을 남깁니다.
- 작성자는 리뷰어의 코멘트에 대해 적극적으로 소통하고 수정합니다.
- **LGTM (Looks Good To Me)** 또는 **Approve**를 받으면 Merge 할 수 있습니다.
